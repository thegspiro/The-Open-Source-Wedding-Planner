# Three User Types Implementation Plan

## Overview

Add user accounts with three distinct roles that shape the UI experience, permissions, and navigation. The app currently has no auth — anyone can see all weddings. This plan adds lightweight authentication and role-based access.

## User Types

1. **Professional Planner** — manages many weddings, sees a multi-wedding dashboard, full access to all features
2. **Helping Friend** — manages 1-2 weddings for someone else, simpler dashboard, same planning tools but lighter UI
3. **Self-Planner** — planning their own single wedding, streamlined experience that goes straight to their wedding dashboard

## Data Model Changes (`models.py`)

### New `User` model:
```python
class User(db.Model):
    id = Column(Integer, primary_key=True)
    email = Column(String(120), unique=True, nullable=False)
    password_hash = Column(String(256), nullable=False)
    name = Column(String(200), nullable=False)
    user_type = Column(String(20), nullable=False)  # 'professional', 'friend', 'self'
    created_at = Column(DateTime, default=datetime.utcnow)

    weddings = relationship('WeddingAccess', backref='user', cascade='all, delete-orphan')
```

### New `WeddingAccess` model (many-to-many with role):
```python
class WeddingAccess(db.Model):
    id = Column(Integer, primary_key=True)
    user_id = Column(Integer, ForeignKey('user.id'), nullable=False)
    wedding_id = Column(Integer, ForeignKey('wedding.id'), nullable=False)
    role = Column(String(20), nullable=False)  # 'owner', 'planner', 'viewer'
    created_at = Column(DateTime, default=datetime.utcnow)
```

### Update `Wedding` model:
- Add `access_list` relationship to `WeddingAccess`

## Auth Routes (`app.py`)

- `GET/POST /register` — registration form with user type selection
- `GET/POST /login` — email + password login
- `GET /logout` — clear session
- Login required decorator for all wedding routes
- Filter weddings by current user's access

## Dependencies

- Add `werkzeug.security` (already available via Flask) for `generate_password_hash` / `check_password_hash`
- No new packages needed — Flask session + werkzeug password hashing is sufficient

## UI Changes

### Registration page (`templates/auth/register.html`)
- Name, email, password fields
- User type selector with descriptions of each type

### Login page (`templates/auth/login.html`)
- Email + password

### Navigation (`templates/base.html`)
- Show logged-in user name
- Login/Logout link
- Hide wedding-specific nav when not in a wedding context

### Home page behavior by user type:
- **Professional**: Multi-wedding dashboard (current `index.html` behavior) with "Add Wedding" button
- **Friend**: Same multi-wedding view but with friendlier language ("Weddings you're helping with")
- **Self-Planner**: Redirect straight to their single wedding dashboard. If no wedding yet, show the "Create Wedding" form directly instead of a list

### Wedding creation:
- **Professional & Friend**: Can create multiple weddings from the index page
- **Self-Planner**: Can only have one wedding. If they already have one, the "new wedding" route redirects to their existing dashboard

## Files to Modify

3. **`templates/auth/register.html`** — New file
4. **`templates/auth/login.html`** — New file

## Key Design Decisions

- **Lightweight auth**: Flask session + werkzeug password hashing. No Flask-Login or Flask-WTF to keep dependencies minimal.
- **No admin role for now**: Professional planner is the closest to admin, but there's no global admin concept.
- **WeddingAccess table**: Enables future sharing (e.g., a self-planner could invite a friend to help). For now, the creator gets 'owner' role automatically.
- **Self-planner limit**: Enforced at the route level (redirect if they already have a wedding), not at the DB level.
- **Existing data migration**: The DB gets recreated via `db.create_all()`. Existing weddings (if any) won't be linked to users, but since this is early-stage that's acceptable.
